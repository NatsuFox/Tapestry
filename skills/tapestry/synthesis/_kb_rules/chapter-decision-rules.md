# Chapter Decision Rules

Use this file when deciding whether to create a new chapter, extend an existing one, or restructure the hierarchy.

## Extend an existing chapter when

- the new feed matches the chapter’s central concept
- the vocabulary and practical use cases align with the chapter
- adding the feed would make the chapter richer rather than blurrier

## Create a new chapter when

- the feed introduces a distinct subdomain that will likely recur
- the existing sibling chapters are all semantically narrower than the new material
- forcing the content into an existing chapter would reduce retrievability

## Split a chapter when

- the chapter now contains more than one clearly different practical use case
- a reader would struggle to predict what belongs there
- its child bullets in the parent `index.md` have become vague or overloaded

## Restructure a topic when

- multiple chapters overlap heavily
- the parent `index.md` has become a list of exceptions instead of a clean map
- historical feeds show two or more stable clusters that deserve different chapter families

## Content-writing rule

When supplementing an existing chapter:

- integrate the new feed into the chapter’s existing narrative
- do not just append a raw dump at the end
- update the chapter’s internal structure if the new material changes the conceptual balance
